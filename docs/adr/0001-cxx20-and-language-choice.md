# ADR-001: Выбор языка программирования и стандарта C++20

## 1. Контекст и проблема

Система предназначена для непрерывного мониторинга на сервере общежития с ограниченными ресурсами. Ключевые требования:
- **Время реакции:** ≤ 5 секунд (от датчика до оповещения).
- **Надежность:** Работа без перезагрузок и деградации производительности в течение длительных периодов.
- **Ресурсы:** Минимальное потребление CPU и RAM (ожидается ~50–100 МБ в idle).

Необходимо выбрать язык программирования для backend-части системы, который обеспечит детерминированное время отклика и минимальный оверхед на оборудовании с ограниченными мощностями.

## 2. Рассмотренные варианты

### Вариант 1: Python

**Описание:** Использование Python 3.10+ с асинхронными фреймворками (FastAPI, aiomqtt, asyncio).
**Плюсы:**
- Быстрая разработка, богатая экосистема библиотек.
- Простота интеграции с ONNX Runtime для инференса.
**Минусы:**
- GIL Блокирует параллельное выполнение CPU-интенсивных задач.
- Недетерминированный GC: сборщик мусора может вызывать непредсказуемые паузы до сотен миллисекунд.
- Высокий memory footprint: Даже "легкий" интерпретатор потребляет 50-100 МБ до загрузки кода.

### Вариант 2: Rust

**Описание:** Использование Rust с async-std или tokio.
**Плюсы:**
- Memory safety без GC.
- Производительность, близкая к C++.
**Минусы:**
- Менее зрелая документация по интеграции специфичных промышленных протоколов.
- Меньше готовых примеров для ONNX Runtime в embedded-сценариях.

### Вариант 3: C++20

**Описание:** Использование современного C++20.
**Плюсы:**
- Отсутствие GC: освобождение памяти детерминированно благодаря RAII.
- Полный контроль над concurrency: явное управление потоками и приоритетами.
- Async I/O: asio позволяет обрабатывать тысячи подключений в event-loop без накладных расходов на переключение контекста ОС.
- Низкий footprint: скомпилированный бинарник не требует интерпретатора.
**Минусы:**
- Риск memory leaks при неаккуратном использовании сырых указателей.

### Вариант 4: Go (Golang 1.22+)

**Описание:** Go с goroutines и ONNX Runtime через CGO.
**Плюсы:**
- Отличная встроенная поддержка concurrency (goroutines + channels) — код читается легко.
- Маленький footprint (обычно 10–25 МБ RSS).
- Простая кросс-компиляция в статический бинарник.
- Быстрая разработка и хорошая экосистема observability.
- **Минусы:**
- Наличие GC.
- Меньший контроль над памятью и временем освобождения ресурсов по сравнению с RAII.
- Интеграция ONNX Runtime требует CGO или менее зрелых Go-bindings.

## 3. Решение

**Выбран C++20:**
- Детерминированное управление ресурсами (RAII, отсутствие GC);
- Минимальный memory/CPU footprint;
- Полный контроль над асинхронностью через Asio;
- Возможность поставки одного компактного бинарного файла.

**Ключевые архитектурные решения, обеспечивающие требования:**
- RAII + `std::unique_ptr`/`shared_ptr` + Core Guidelines для безопасной работы с памятью.
- Библиотека `asio` для асинхронного I/O.
- Статическая линковка всех зависимостей.
- Строгий режим компиляции (`-Wall -Wextra -Werror -Wpedantic`).
