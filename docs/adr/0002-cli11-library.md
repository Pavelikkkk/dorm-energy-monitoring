# ADR-002: Интерфейс командной строки (CLI) для управления сервисом

## 1. Контекст и проблема

Система работает как фоновый демон на сервере общежития 24/7. Администраторам и разработчикам необходим удобный способ взаимодействия с сервисом для:
1.  **Управления жизненным циклом:** запуск, остановка, перезагрузка, проверка статуса.
2.  **Отладки и мониторинга:** просмотр последних логов, ручное тестирование компонентов (MQTT, БД, ИИ).
3.  **Конфигурации:** валидация конфигов перед перезапуском, генерация дефолтных настроек.

Требования к CLI:
-   **Нативность:** Должен быть частью единого бинарного файла или поставляться вместе с ним без дополнительных зависимостей.
-   **Производительность:** Мгновенный запуск.
-   **UX:** Поддержка автодополнения, человеко-читаемый вывод, цветовая подсветка.
-   **Безопасность:** Не должен требовать root-прав для чтения статуса, но может требовать их для управления сервисом.

## 2. Рассмотренные варианты

### Вариант 1: Использование системных утилит (`systemctl`, `journalctl`) + простой флаг `--help`
*   **Описание:** Полагаться исключительно на `systemd` для управления демоном. Внутри бинаря реализовать только минимальный парсинг аргументов (`getopt` или `std::format` для вывода версии).
*   **Плюсы:** Нет лишних зависимостей.
*   **Минусы:** 
    -   Сложная отладка: нужно знать команды `journalctl -u service_name -f`.
    -   Нет встроенных инструментов для быстрой проверки связи с MQTT/БД без написания отдельных скриптов.
    -   Плохой UX для разработчиков (нет интерактивных команд типа `cli test mqtt`).

### Вариант 2: Самописный парсер аргументов
*   **Описание:** Написать свой парсер на основе `argc/argv` или простых библиотек вроде `cxxopts`.
*   **Плюсы:** Полный контроль, нулевые зависимости.
*   **Минусы:** 
    -   Высокие затраты времени на реализацию подкоманд, справки, валидации.
    -   Отсутствие генерации shell-completion из коробки.
    -   Риск получить неудобный и неподдерживаемый интерфейс.

### Вариант 3: Библиотека `CLI11`
*   **Описание:** Использование современной header-only библиотеки `CLI11`.
*   **Плюсы:** 
    -   Богатый функционал: подкоманды, группы опций, валидация типов, генерация справки.
    -   Интеграция с `fmt`/`std::format` для красивого вывода.
    -   Header-only: легко добавить в проект через vcpkg или git submodule.
*   **Минусы:** 
    -   Дополнительная зависимость (хотя и легковесная).
    -   Чуть больший размер бинарника.

## 3. Решение

Использовать библиотеку **CLI11** для реализации встроенного интерфейса командной строки.

Бинарный файл будет поддерживать два режима работы:
1.  **Daemon Mode (по умолчанию):** Запуск основного цикла событий.
2.  **CLI Mode (при наличии флагов/подкоманд):** Выполнение утилитарных задач и выход.
