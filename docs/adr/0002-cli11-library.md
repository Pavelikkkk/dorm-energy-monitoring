# ADR-002: Интерфейс командной строки (CLI) для управления сервисом

## 1. Контекст и проблема

Система работает как фоновый демон (`systemd`-сервис) на сервере общежития 24/7. Администраторам и разработчикам нужен удобный и надёжный способ взаимодействия с сервисом без необходимости писать отдельные скрипты.

**Требования к CLI:**
- Поддержка управления жизненным циклом (start/stop/restart/status).
- Инструменты отладки и мониторинга (тестирование MQTT, БД, генерация синтетических данных).
- Валидация конфигурации и генерация дефолтных настроек.
- Нативность: один бинарный файл, минимум зависимостей.
- UX: автодополнение, `--help`, цветовой вывод, подкоманды.
- Производительность: мгновенный запуск в CLI-режиме.

## 2. Рассмотренные варианты

### Вариант 1: Только systemd + journalctl

**Описание:** Полагаться на `systemctl` и `journalctl`, внутри бинарника — минимальный `--help`.
**Плюсы:** 
- Нет лишних зависимостей.
**Минусы:** 
- Сложная отладка.
- Нет встроенной валидации конфигов.
- Плохой UX для разработчиков (нет интерактивных команд типа `cli test mqtt`).

### Вариант 2: Самописный парсер аргументов

**Описание:** Написать свой парсер на основе `argc/argv` или простых библиотек вроде `cxxopts`.
**Плюсы:** Полный контроль, нулевые зависимости.
**Минусы:** 
- Высокие затраты на разработку подкоманд, валидации, справки и completion.
- Легко допустить ошибки (особенно с UTF-8 и сложными опциями).
- Плохо масштабируется при росте проекта.

### Вариант 3: Библиотека `CLI11`

**Описание:** Использование современной header-only библиотеки `CLI11`.
**Плюсы:** 
-   Богатый функционал: подкоманды, группы опций, валидация типов, генерация справки.
-   Интеграция с `fmt`/`std::format` для красивого вывода.
-   Header-only: легко добавить в проект через vcpkg или git submodule.
**Минусы:** 
-   Дополнительная зависимость (хотя и легковесная).
-   Чуть больший размер бинарника.

## 3. Решение

**Использовать библиотеку CLI11.**

**Почему CLI11:**
- Лучшее сочетание «мощь / вес / удобство» среди всех рассмотренных вариантов.
- Полностью соответствует требованиям к UX и нативности.
- Минимально влияет на размер бинарника.

**Архитектурные решения:**
- Бинарный файл `dorm-sim` поддерживает два режима:
  - **Daemon mode** (по умолчанию) — запускает основной event-loop.
  - **CLI mode** — при наличии подкоманд/флагов выполняет утилитарные задачи и завершается.
- Все зависимости (включая CLI11) собираются статически.
- Используется `vcpkg` (или git submodule) для управления зависимостями.
