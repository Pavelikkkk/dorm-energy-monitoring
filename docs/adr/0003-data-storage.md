# ADR-003: Архитектура хранения данных и выбор СУБД

## 1. Контекст и проблема

Система имитации и мониторинга энергопотребления обрабатывает непрерывный поток телеметрии от IoT-датчиков ( движение, потребляемая мощность) на сервере общежития с ограниченными ресурсами.

**Требования к хранению данных:**
- **Высокая интенсивность записи:** Поддержка до 1000+ событий/сек в пиковые часы без потери данных.
- **Эффективность хранения:** Автоматическое управление жизненным циклом данных — удаление старых "сырых" записей и агрегация исторических данных для экономии дискового пространства.
- **Скорость аналитики:** Быстрое построение отчётов за месяцы и годы без сканирования миллионов сырых строк (время ответа < 100 мс).
- **Надёжность:** Гарантированная сохранность данных при внезапных сбоях питания.
- **Интеграция с C++:** Наличие зрелой, документально оформленной библиотеки для асинхронной работы, пулов соединений и транзакций.
- **Универсальность:** Возможность хранить как временные ряды (телеметрия датчиков), так и реляционные данные (пользователи, настройки алертов, метаданные устройств) в единой системе.

## 2. Рассмотренные варианты

### Вариант 1: Чистый PostgreSQL
- **Описание:** Использование стандартного PostgreSQL с ручным управлением партиционированием, агрегацией и очисткой старых данных.
- **Плюсы:**
    - Максимально надёжная и стабильная СУБД.
    - Широкая поддержка инструментами.
    - Зрелая C++ библиотека `libpqxx`.
- **Минусы:**
    - Удаление старых данных через `DELETE` фрагментирует таблицу и требует тяжёлого `VACUUM`, создавая высокую нагрузку на CPU/диск.
    - Нет встроенных инструментов для автоматической агрегации временных рядов.

### Вариант 2: PostgreSQL + TimescaleDB
- **Описание:** Использование PostgreSQL с расширением TimescaleDB, добавляющим гипертаблицы, непрерывные агрегаты и политики автоматического удаления старых данных.
- **Плюсы:**
    - **Автоматическое партиционирование:** Гипертаблицы автоматически разбивают данные по времени, обеспечивая высокую скорость записи и эффективные индексы.
    - **Continuous Aggregates:** База данных сама рассчитывает и хранит агрегаты (среднее/мин/макс за час/день), ускоряя долгосрочные отчёты.
    - **Retention Policies:** Автоматическое удаление старых сырых данных путём сброса целых партиций (мгновенно и без фрагментации).
    - **Единый источник истины:** Хранит и временные ряды, и реляционные данные, позволяя делать сложные `JOIN` запросы.
    - **ACID и WAL:** Полная надёжность стандартного PostgreSQL.
- **Минусы:**
    - Дополнительное расширение.

### Вариант 3: SQLite
- **Описание:** Файловая база данных, работающая внутри процесса C++ без отдельного сервера.
- **Плюсы:**
    - Нулевая конфигурация, не требует отдельного сервиса.
    - Очень маленький footprint (< 1 МБ RAM).
    - Простота бэкапа (один файл).
- **Минусы:**
    - **Проблемы с конкурентной записью:** Один пишущий поток блокирует всю базу.
    - **Отсутствие автоматического downsampling:** Нет встроенных механизмов агрегации старых данных.
    - **Удаление старых данных:** `DELETE` требует `VACUUM`, который переписывает весь файл.
    - **Ограниченная аналитика:** Сложные отчёты за годы работают медленно без серьёзной денормализации схемы.

### Вариант 4: InfluxDB
- **Описание:** Специализированная time-series база данных.
- **Плюсы:**
    - Оптимизирована для высокоинтенсивной записи временных рядов.
    - Встроенные retention policies и downsampling.
    - Эффективное сжатие данных.
- **Минусы:**
    - **Требует запуска отдельного сервиса**.
    - **Язык запросов Flux сложнее в поддержке**, чем SQL.
    - **Сложная интеграция с реляционными данными** (пользователи, настройки алертов) — потребовалась бы вторая БД.
    - **C++ клиенты** менее зрелые, чем `libpqxx`, и часто требуют обёрток над HTTP API.

### Вариант 5: Файловые системы (CSV/JSON + ручная индексация)
- **Описание:** Прямая запись в CSV-файлы или JSON-логи с ручным управлением ротацией.
- **Плюсы:**
    - Максимальная простота, нулевые зависимости.
    - Легко читать внешними инструментами.
- **Минусы:**
    - **Отсутствие индексов:** Любой аналитический запрос требует полного сканирования всех файлов.
    - **Риск повреждения при сбоях:** Нет журнала предзаписи, при внезапном отключении питания можно потерять последние записи.
    - **Ручное управление агрегацией и ротацией:**.

## 3. Решение

Использовать **PostgreSQL** с расширением **TimescaleDB** в качестве единого хранилища данных.

**Схема хранения:**
1.  **Гипертаблица для сырых событий:** Хранит телеметрию датчиков с Retention Policy на 30 дней.
2.  **Continuous Aggregates:** Автоматически пересчитываемые часовые и суточные агрегаты для долгосрочных отчётов.
3.  **Обычные таблицы PostgreSQL:** Хранят реляционные данные (устройства, пользователи, правила алертов).

**Коммуникация с C++:** Через библиотеку `libpqxx` (асинхронные запросы, пулы соединений, транзакции).
