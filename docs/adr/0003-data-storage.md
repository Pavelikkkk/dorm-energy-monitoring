# ADR-003: Архитектура хранения данных и выбор СУБД

## 1. Контекст и проблема

Система обрабатывает непрерывный поток телеметрии от IoT-датчиков (до 1000+ событий/сек в пике). Данные должны храниться долго, быстро записываться и эффективно анализироваться.

**Ключевые требования:**
- Высокая интенсивность записи и надёжность (гарантия доставки даже при сбоях питания).
- Автоматическое управление жизненным циклом данных (удаление старых записей без фрагментации).
- Быстрая аналитика за месяцы/годы (< 100 мс).
- Поддержка как time-series, так и реляционных данных (устройства, пользователи, правила алертов) в одной системе.
- Хорошая, зрелая C++-библиотека (асинхронные запросы, пулы, транзакции).

### Вариант 1: Чистый PostgreSQL

**Плюсы:** Максимальная стабильность, отличная C++-интеграция (`libpqxx`).  
**Минусы:** Тяжёлый `VACUUM` при удалении старых данных, нет встроенных инструментов для time-series.

### Вариант 2: PostgreSQL + TimescaleDB
**Плюсы:**
- Гипертаблицы + автоматическое партиционирование.
- Continuous Aggregates (автоматические materialized views).
- Retention Policies (мгновенное удаление целых партиций).
- Единая БД для time-series и реляционных данных.
- Полная ACID-надёжность PostgreSQL.

**Минусы:** Дополнительное расширение.

### Вариант 3: SQLite
**Плюсы:** Нулевая конфигурация, минимальный footprint.  
**Минусы:** Проблемы с конкурентной записью, тяжёлый `VACUUM`, слабая аналитика.

### Вариант 4: InfluxDB
**Плюсы:** Отличная оптимизация под time-series.  
**Минусы:** Отдельный сервис, сложный Flux, проблемы с реляционными данными, слабая C++-интеграция.

### Вариант 5: ClickHouse
**Описание:** Колоночная OLAP-СУБД, отлично подходящая для высоконагруженных time-series.  
**Плюсы:** Быстрая запись и аналитика, эффективное сжатие, materialized views.  
**Минусы:** Слабее в транзакциях и реляционных JOIN’ах, отдельный сервис, менее зрелая C++-библиотека по сравнению с `libpqxx`.

### Вариант 6: Файловая система (CSV / custom binary)
**Минусы:** Отсутствие индексов, ACID, сложность аналитики и бэкапов.

## 3. Решение

**Выбран PostgreSQL + TimescaleDB.**

**Почему именно это сочетание:**
TimescaleDB решает все основные боли чистого PostgreSQL в сценариях IoT, сохраняя при этом все преимущества знакомой SQL и отличной C++-интеграции. ClickHouse был бы быстрее для чистой аналитики, но проигрывает в единстве хранения и надёжности транзакций.

**Конкретные архитектурные решения:**
- Гипертаблица для сырых измерений (retention policy 30–90 дней).
- Continuous Aggregates для часовых/суточных отчётов.
- Обычные PostgreSQL-таблицы для устройств, пользователей, настроек алертов.
- Доступ через `libpqxx` (пулы соединений, prepared statements, асинхронные запросы).
