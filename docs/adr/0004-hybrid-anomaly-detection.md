# ADR-004: Гибридная детекция аномалий (Пороговые правила + ML)

## 1. Контекст и проблема

Система мониторинга энергопотребления должна выявлять нештатные ситуации в режиме реального времени на основе данных с датчиков.

**Требования к детекции:**
- **Низкая задержка:** Время от возникновения аномалии до оповещения ≤ 5 секунд.
- **Минимум ложных срабатываний:** Система не должна беспокоить администратора из-за кратковременных шумов или сбоев датчиков.
- **Интерпретируемость:** Администратор должен понимать, *почему* сработала тревога.
- **Ресурсоэффективность:** Инференс ML-модели не должен блокировать основной поток обработки MQTT на слабом железе.

## 2. Рассмотренные варианты

### Вариант 1: Только пороговые правила
- **Описание:** Использование детерминированных правил вида с фильтрацией шумов через скользящее среднее.
- **Плюсы:**
    - Мгновенная реакция =
    - Полная интерпретируемость
    - Не требует обучения, легко настраивается
    - Работает даже при полной загрузке CPU
- **Минусы:**
    - **Высокий уровень ложных срабатываний** при кратковременных выбросах (даже с фильтрацией)
    - **Невозможность детектировать сложные паттерны**
    - **Отсутствие контекста** 

### Вариант 2: Только ML-модель
- **Описание:** Передача всех данных в нейросеть, которая классифицирует состояние как «норма» или «аномалия».
- **Плюсы:**
    - Высокая точность на сложных, многомерных паттернах
    - Автоматическая адаптация к новым данным через дообучение
- **Минусы:**
    - **Вычислительная нагрузка:** Постоянный инференс может не уложиться в 5 секунд на слабом CPU
    - **«Слепое пятно»:** При падении или зависании модели детекция полностью прекращается
    - **Сложность интерпретации:** 
    - **Требует буферизации:** Для контекстного анализа нужна история за N минут

### Вариант 3: Облачная аналитика
- **Описание:** Отправка всех данных в облачный сервис для детекции.
- **Плюсы:**
    - Неограниченные вычислительные ресурсы
    - Готовые ML-пайплайны
- **Минусы:**
    - **Зависимость от интернета:** При обрыве связи детекция отсутствует
    - **Латентность:** Задержка может достигать минут

### Вариант 4: Гибридная детекция (Пороговые правила + ML)
- **Описание:** Двухуровневая архитектура:
    - **Level 1 (Fast Path):** Детерминированные пороговые правила с фильтрацией шумов
    - **Level 2 (Smart Path):** ML-модель для сложных паттернов, выполняемая асинхронно
- **Плюсы:**
    - **Безопасность (Safety Net):** Даже при падении ML пороговые правила продолжают работать
    - **Быстрая реакция на критические события:** Пороги срабатывают мгновенно
    - **Снижение ложных срабатываний:** ML отсеивает шумы, которые пороги ловят ложно
    - **Экономия ресурсов:** ML инференс только для «подозрительных» случаев (или асинхронно в фоне)
- **Минусы:**
    - **Сложность архитектуры:** Нужно управлять двумя конвейерами детекции
    - **Дополнительная память для буфера истории**

## 3. Решение

Использовать **гибридную двухуровневую архитектуру детекции аномалий**:

### Уровень 1: Пороговые правила (Fast Path)
- **Hard Limits:** Абсолютные пороги для критических датчиков
- **Фильтрация шумов:** Экспоненциальное скользящее среднее для отсечения выбросов
- **Дедюпликация:** Подавление повторных алертов в течение окна (например, 5 минут)

### Уровень 2: ML-модель (Smart Path)
- **Платформа:** ONNX Runtime (инференс внутри процесса C++)
- **Вход модели:** Вектор признаков (последние N значений с датчиков)
- **Выход:** Вероятность аномалии (0..1)
- **Триггер:** Алерт при вероятности > порога (например, 0.85)

### Логика комбинирования (OR)
Тревога генерируется, если сработал **хотя бы один** из уровней. Для критических событий (пожар) порог Level 1 намеренно занижен, чтобы обеспечить 100% обнаружение.

### Изоляция потоков
- **Поток 1 (MQTT Handler):** Приём данных, проверка Level 1, отправка в буфер истории
- **Поток 2 (ML Worker):** Асинхронный инференс ONNX модели, не блокирует поток 1
