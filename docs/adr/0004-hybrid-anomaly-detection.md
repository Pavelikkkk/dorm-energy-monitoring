# ADR-004: Гибридная детекция аномалий (Пороговые правила + ML)

## 1. Контекст и проблема

Система мониторинга энергопотребления должна выявлять нештатные ситуации в режиме реального времени на основе данных с датчиков.

**Ключевые требования:**
- Задержка от возникновения аномалии до оповещения ≤ 5 секунд.
- Минимальное количество ложных срабатываний.
- Высокая интерпретируемость (администратор должен понимать «почему» сработала тревога).
- Ресурсоэффективность на слабом железе.
- Надёжность: система не должна «слепнуть» при отказе ML-модели.

## 2. Рассмотренные варианты

### Вариант 1: Только пороговые правила

**Описание:** Использование детерминированных правил вида с фильтрацией шумов через скользящее среднее.
**Плюсы:**
- Мгновенная реакция =
- Полная интерпретируемость
- Не требует обучения, легко настраивается
- Работает даже при полной загрузке CPU
**Минусы:**
- Высокий уровень ложных срабатываний при кратковременных выбросах (даже с фильтрацией)
- Невозможность детектировать сложные паттерны
- Отсутствие контекста 

### Вариант 2: Только ML-модель

**Описание:** Передача всех данных в нейросеть, которая классифицирует состояние как «норма» или «аномалия».
**Плюсы:**
- Высокая точность на сложных, многомерных паттернах
- Автоматическая адаптация к новым данным через дообучение
**Минусы:**
- Вычислительная нагрузка: постоянный инференс может не уложиться в 5 секунд на слабом CPU
- При падении или зависании модели детекция полностью прекращается
- Сложность интерпретации 
- Требует буферизации: для контекстного анализа нужна история за N минут

### Вариант 3: Облачная аналитика

**Описание:** Отправка всех данных в облачный сервис для детекции.
**Плюсы:**
- Неограниченные вычислительные ресурсы
- Готовые ML-пайплайны
**Минусы:**
- Зависимость от интернета: при обрыве связи детекция отсутствует
- Задержка может достигать минут

### Вариант 4: Гибридная детекция (Пороговые правила + ML)

**Описание:** Двухуровневая архитектура:
- Level 1 (Fast Path): Детерминированные пороговые правила с фильтрацией шумов
- Level 2 (Smart Path): ML-модель для сложных паттернов, выполняемая асинхронно
**Плюсы:**
- Safety Net: пороговые правила работают даже при падении ML.
- Мгновенная реакция на критические события.
- Снижение ложных срабатываний за счёт ML-фильтрации.
- Экономия ресурсов (ML запускается только для подозрительных случаев или асинхронно).
**Минусы:**
- Увеличение сложности архитектуры.
- Дополнительная память для буфера истории.

## 3. Решение

**Выбрана двухуровневая гибридная архитектура.**

**Level 1 (Fast Path — пороговые правила):**
- Hard limits + экспоненциальное скользящее среднее (EMA) для фильтрации шумов.
- Дедюпликация алертов.
- Выполняется синхронно в потоке MQTT Handler.

**Level 2 (Smart Path — ML-модель):**
- ONNX Runtime (C++ API) — асинхронный worker.
- Вход: вектор последних N измерений.
- Выход: вероятность аномалии.
- Триггер при вероятности > 0.85 (настраиваемо).

**Логика комбинирования:** OR (тревога, если сработал любой уровень).  
Для критических событий (например, пожар) пороги Level 1 намеренно занижены, чтобы обеспечить 100% recall.
