# ADR-005: Генератор синтетических данных для разработки и тестирования

## 1. Контекст и проблема

Для разработки и автономного тестирования системы мониторинга необходим источник данных, имитирующий реальные сенсоры. Требования:

**Ключевые требования:**
- Полная независимость от реального оборудования и MQTT-брокера.
- Воспроизводимость (детерминированные данные при фиксированном seed).
- Возможность эмуляции различных сценариев аномалий.
- Низкое потребление ресурсов.
- Лёгкая интеграция с существующим CLI и AnomalyDetector.

Отсутствие генератора приводит к:
- Невозможности тестирования edge-кейсов без реального оборудования.
- Flaky тестам при использовании случайных данных.
- Зависимости разработчиков от доступности физических датчиков.

## 2. Рассмотренные варианты

### Вариант 1: Запись и воспроизведение реального трафика

**Описание:** Логирование реальных MQTT-сообщений от ESP32 в файл и последующее воспроизведение в тестовой среде.
**Плюсы:**
- Абсолютно реалистичные данные.
- Не требует написания генератора.
**Минусы:**
- Не позволяет генерировать новые сценарии аномалий.
- Нельзя изменять частоту событий без перезаписи логов.
- Зависимость от предварительно собранного корпуса данных.

### Вариант 2: Python-скрипт-генератор

**Описание:** Отдельный Python-скрипт, публикующий синтетические данные в MQTT брокер.
**Плюсы:**
- Быстрая реализация.
- Гибкость (легко добавить новые сценарии).
**Минусы:**
- Накладные расходы на межпроцессное взаимодействие.
- Усложнение CI.
- Недетерминированная производительность.

### Вариант 3: Простая случайная генерация внутри core

**Описание:** Функция внутри core-логики, генерирующая случайные показания датчиков в рантайме.
**Плюсы:**
- Минимальная сложность реализации.
**Минусы:**
- Результаты меняются от запуска к запуску.
- Невозможность воспроизвести конкретный баг.
- Случайные данные могут скрыть проблемы валидации.

### Вариант 4: Отдельный C++20 модуль с детерминированной генерацией 

**Описание:** Компилируемый опционально C++20 модуль, эмулирующий MQTT-клиент. Генератор принимает seed и конфигурационный файл со сценариями.
**Плюсы:**
- Полная изоляция от core-логики.
- Детерминизм при фиксированном seed.
- Возможность эмулировать 100+ сенсоров с точным соблюдением периодов опроса.
- В production бинар не попадает.
**Минусы:**
- Дополнительная работа по реализации.

## 3. Решение

**Выбран встроенный C++20 модуль `SyntheticDataGenerator`.**

**Почему именно этот вариант:** 
Это позволяет уже на ранних итерациях полноценно тестировать всю цепочку (генерация → детекция → хранение), не дожидаясь реального оборудования.

**Ключевые технические решения:**
- `GeneratorConfig` с параметрами `days`, `seed`, `inject_anomalies`.
- Детерминированная генерация на основе `std::mt19937` и `std::normal_distribution`.
- Интеграция с CLI (`simulate` subcommand).
- Возможность расширения (добавление новых типов датчиков, паттернов потребления).
