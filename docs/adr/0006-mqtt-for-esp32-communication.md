# ADR-006: Протокол взаимодействия с ESP32

## 1. Контекст и проблема

Система мониторинга общежития использует ESP32 в качестве конечных устройств сбора данных с датчиков. Требования к протоколу связи между ESP32 и бэкендом:

- **Энергоэффективность:** ESP32 может работать от батарей, протокол должен минимизировать потребление энергии.
- **Надёжность доставки:** Потеря пакетов в Wi-Fi сети не должна приводить к потере критических событий.
- **Низкий оверхед:** Ограниченные ресурсы ESP32 требуют компактных сообщений.
- **Асинхронность:** Устройства отправляют данные без необходимости поддерживать постоянное соединение.
- **Масштабируемость:** Поддержка до 100+ устройств в одной сети общежития.

Необходимо выбрать протокол, который удовлетворит все эти требования без излишнего усложнения прошивки.

## 2. Рассмотренные варианты

### Вариант 1: HTTP/REST 

- **Описание:** ESP32 отправляет HTTP POST запросы с данными датчиков на backend-сервер.
- **Плюсы:**
    - Стандартный протокол, легко отлаживать.
    - Широкая поддержка на бэкенде.
- **Минусы:**.
    - Каждое измерение требует нового TCP соединения.
    - Нет встроенной ретрансляции при потере пакетов.
    - Потребление энергии на ESP32 в 5–10 раз выше, чем у MQTT.

### Вариант 2: CoAP

- **Описание:** UDP-based протокол, специально разработанный для constrained устройств.
- **Плюсы:**
    - Легковесный.
    - Работает поверх UDP, что снижает overhead.
    - Встроенная поддержка подтверждений и ретрансляции.
- **Минусы:**
    - Требует реализации подтверждений поверх UDP.
    - Брокер для CoAP менее зрелый, чем Mosquitto.
    - Меньше готовых библиотек для ESP32.
    - Сложнее в отладке.

### Вариант 3: Кастомный UDP-протокол

- **Описание:** Разработать свой примитивный протокол поверх UDP (например, последовательность байт: `sensor_id | value | timestamp`).
- **Плюсы:**
    - Минимальный оверхед (8–16 байт на сообщение).
    - Полный контроль.
- **Минусы:**
    - Нужно изобретать механизмы подтверждения, ретрансляции, фрагментации.
    - Риск получить нестабильную реализацию.
    - Отсутствие готовых инструментов мониторинга.
    - Высокие затраты на разработку и отладку.

### Вариант 4: MQTT поверх TCP/IP 

- **Описание:** ESP32 выступает в роли MQTT-клиент.
- **Плюсы:**
    - **Энергоэффективность:** Заголовок MQTT — 2 байта, поддержка deep sleep между публикациями.
    - **Масштабируемость:** Один брокер Mosquitto выдерживает тысячи клиентов.
    - **Готовые библиотеки для ESP32:** PubSubClient, EspMQTTClient.
    - **Зрелая экосистема:** Инструменты мониторинга, отладки, брокеры с persistency.
- **Минусы:**
    - TCP overhead выше, чем у UDP.
    - Требуется поддерживать брокер.

## 3. Решение

Использовать **MQTT поверх TCP/IP** с QoS уровнями 0 и 1. ESP32 — MQTT-клиент, бэкенд — MQTT брокер.
