# ADR-008: Инференс ML-модели (ONNX Runtime)

## 1. Контекст и проблема

Система мониторинга должна выполнять классификацию событий на основе данных с датчиков и, возможно, изображений с камер. Требования к подсистеме инференса:

- **Время реакции:** Полный цикл от получения данных до принятия решения не должен превышать 2 секунд.
- **Детерминизм:** Время выполнения инференса должно быть предсказуемым, без непредвиденных пауз.
- **Гибкость:** Возможность подмены модели без перекомпиляции всей системы.
- **Совместимость:** Модель должна тренироваться на отдельной инфраструктуре и деплоиться в C++ приложение.

## 2. Рассмотренные варианты

### Вариант 1: Прямой вызов Python-модели через subprocess или embedding Python

- **Описание:** Запуск Python-скрипта с моделью через subprocess или встраивание интерпретатора Python в C++.
- **Плюсы:**
    - Полный доступ к экосистеме Python ML-библиотек.
    - Можно использовать обученную модель без конвертации.
- **Минусы:**
    - **Недетерминизм:** Python GC может вызвать паузы.
    - **Overhead:** Сериализация данных между C++ и Python.
    - **Высокий memory footprint:** Интерпретатор Python + модель.
    - **Сложность деплоя:** Нужно доставлять Python-окружение и зависимости.

### Вариант 2: TensorFlow Lite C++ API

- **Описание:** Конвертация модели в TensorFlow Lite и инференс через нативный C++ API.
- **Плюсы:**
    - Легковесный рантайм.
    - Поддержка оптимизаций.
    - Хорошая производительность на CPU.
- **Минусы:**
    - Модель нужно конвертировать из PyTorch/TensorFlow в TFLite.
    - Меньшее количество готовых примеров для embedded-сценариев.
    - Экосистема C++ API менее удобна, чем ONNX Runtime.

### Вариант 3: ONNX Runtime

- **Описание:** Использование ONNX Runtime — кросс-платформенного inference engine от Microsoft, поддерживающего ONNX-формат.
- **Плюсы:**
    - **Универсальность:** Принимает модели из PyTorch, TensorFlow, и других.
    - **Производительность:** Оптимизации для CPU, поддержка quantization для ускорения.
    - **Низкий overhead:** Написан на C++, без GC.
    - **Детерминизм:** Память выделяется заранее.
- **Минусы:**
    - Дополнительная зависимость.
    - Требуется конвертация модели в ONNX .

### Вариант 4: libtorch

- **Описание:** Использование нативного PyTorch C++ API (libtorch) для загрузки и инференса TorchScript-модели.
- **Плюсы:**
    - Прямой экспорт из PyTorch без промежуточного ONNX.
    - Богатая функциональность.
- **Минусы:**
    - **Большой размер:** libtorch весит ~200 МБ.
    - **Высокий memory footprint.**
    - Более сложная сборка и линковка.
    - Избыточно для задач классификации временных рядов и простых CNN.

## 3. Решение

Использовать **ONNX Runtime C++ API** в качестве inference engine для всех ML-моделей в системе.

### Архитектурные решения:

- **Предобучение:** Модели тренируются на Python (PyTorch или TensorFlow), затем экспортируются в ONNX.
