# Функциональное тестирование

Функциональное тестирование проводилось для проверки основных пользовательских и системных сценариев программной системы мониторинга энергопотребления общежития.

Для автоматизации процесса тестирования использовался фреймворк Google Test. Дополнительно применялись интеграционные проверки REST API, PostgreSQL/TimescaleDB, MQTT-компонентов и нагрузочный сценарий k6. В связи с использованием автоматизированного тестирования вместо подробного описания каждого тест-кейса приведены наиболее значимые тестовые сценарии.

## TC-01. Регистрация пользователя

Цель теста: проверить возможность создания нового пользователя в системе.

Предусловия:
- серверная часть приложения запущена;
- база данных PostgreSQL/TimescaleDB доступна;
- пользователь с указанным email отсутствует в системе.

Шаги выполнения:
1. Отправить POST-запрос на endpoint `/api/auth/register`.
2. Передать имя пользователя, email, пароль и тип аккаунта.
3. Проверить ответ сервера.

Ожидаемый результат:
- сервер возвращает признак успешного выполнения;
- в ответе присутствует идентификатор созданного пользователя;
- данные пользователя сохраняются в базе данных.

Автоматизированная проверка:
- `AuthServiceTest.RegistersPersonalUserWithHashedPassword`;
- `AuthServiceTest.RegistersBusinessAccountWhenRequested`;
- интеграционная проверка REST API при выполнении k6-сценария.

Результат: выполнено успешно.

## TC-02. Авторизация пользователя

Цель теста: проверить возможность входа пользователя в систему с корректными учетными данными.

Предусловия:
- пользователь зарегистрирован в системе;
- пароль пользователя сохранен в хешированном виде.

Шаги выполнения:
1. Отправить POST-запрос на endpoint `/api/auth/login`.
2. Передать email и пароль пользователя.
3. Проверить наличие JWT-токена в ответе.

Ожидаемый результат:
- при корректных учетных данных сервер возвращает JWT-токен;
- при неверном пароле сервер возвращает ошибку авторизации.

Автоматизированная проверка:
- `AuthServiceTest.LoginReturnsGeneratedToken`;
- `AuthServiceTest.LoginRejectsMissingUserAndWrongPassword`;
- live-проверка REST API: неверный пароль возвращает HTTP `401`.

Результат: выполнено успешно.

## TC-03. Проверка JWT-токена

Цель теста: проверить механизм валидации JWT-токена при доступе к защищенным ресурсам.

Предусловия:
- пользователь успешно авторизован;
- получен JWT-токен.

Шаги выполнения:
1. Выполнить запрос к защищенному endpoint `/api/account` без токена.
2. Выполнить запрос с некорректным JWT-токеном.
3. Выполнить запрос с просроченным JWT-токеном.
4. Выполнить запрос с корректным JWT-токеном.

Ожидаемый результат:
- запрос без токена возвращает HTTP `401`;
- запрос с некорректным токеном возвращает HTTP `401`;
- запрос с просроченным токеном возвращает HTTP `401`;
- запрос с корректным токеном выполняется успешно.

Автоматизированная проверка:
- `AuthInfrastructureTest.JwtServiceGeneratesAndValidatesClaims`;
- `AuthInfrastructureTest.JwtServiceRejectsMalformedAndWronglySignedTokens`;
- `AuthServiceTest.ValidateTokenDelegatesToJwtService`;
- live-проверка REST API защищенных endpoint.

Результат: выполнено успешно.

## Команды запуска автоматизированных проверок

Запуск модульных тестов:

```powershell
ctest --test-dir build -C Debug --output-on-failure
```

Запуск интеграционных тестов PostgreSQL/TimescaleDB:

```powershell
$env:DORM_ENERGY_TEST_DATABASE_URL='host=localhost port=5432 dbname=dorm_energy user=postgres password=postgres'
ctest --test-dir build -C Debug --output-on-failure -R PostgresRepositoryIntegrationTest
```

Запуск нагрузочного теста REST API через k6:

```powershell
docker run --rm -e BASE_URL=http://host.docker.internal:8080/api -v "${PWD}\tests\load:/scripts" grafana/k6 run /scripts/rest-api.k6.js
```
